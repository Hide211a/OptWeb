# Сценарій захисту диплому (7–10 хв)

## Підготовка

```bash
cd backend && npm run db:setup
cd frontend && npm run dev
```

Демо-акаунти: `manager@optsklad.ua`, `director@optsklad.ua` / `demo123`

Після seed у системі є:
- **2 прострочені партії** (кефір, куряче філе) — для списання
- **1 партія «термін ≤ 7 днів»** (сметана) — для робочого столу
- Проведені надходження, реалізація, чернетка відвантаження

---

## Крок 1 — Менеджер, робочий стіл (1 хв)

1. Увійти як **manager@optsklad.ua**
2. Відкрити **Робочий стіл**
3. Показати чергу задач: чернетки, низький залишок, **«Термін ≤ 7 днів»**, **прострочені партії**
4. Панель **сповіщень** (Alert) з посиланнями на партії / документи
5. Перейти на **Партії** → фільтр прострочення або зони

**Що пояснити:** опт продовольства = партійний облік + термін придатності, не як retail з IMEI.

---

## Крок 2 — FEFO та реалізація (2 хв)

1. **Документи** → чернетка **ВН-2026-0001** або створити нову **Реалізацію**
2. Майстер → крок **Товари** → крок **Підтвердження**
3. Показати **План списання FEFO** (з яких партій піде товар)
4. **Провести** документ

**Що пояснити:** FEFO = First Expired, First Out; алгоритм у `docs/FEFO.md`, тести `npm test` у backend.

---

## Крок 3 — Списання прострочки (1,5 хв)

1. **Документи** → **Створити** → **Списання**
2. Панель **«Прострочено на складі»** → **Додати** партію
3. Провести списання

**Що пояснити:** зменшення втрат, контроль прострочення для оптової бази.

---

## Крок 4 — Надходження з терміном (1 хв)

1. Створити **Надходження**
2. На рядку вказати **термін придатності** (дата з маркування партії постачальника)
3. Зберегти чернетку / провести

**Що пояснити:** реальний бізнес-процес — термін не завжди = дата документа + shelfLifeDays.

---

## Крок 5 — B2B-клієнт і прайс-лист (1,5 хв)

1. **Клієнти** → картка (напр. «Свіжий ранок»)
2. Блок **Прайс-лист** — індивідуальні ціни на товари
3. **Документи** → **Реалізація** → обрати цього клієнта → ціни підставляються автоматично

**Що пояснити:** B2B-ціноутворення окремо для кожного оптового клієнта.

---

## Крок 6 — Керівник, аналітика та аудит (2 хв)

1. Вийти → **director@optsklad.ua**
2. **Аналітика** — KPI, графік реалізації, топ клієнтів
3. **Звіти** → **Прострочення** / **Рух** / **Топ клієнтів** → **CSV** або **PDF**
4. **Журнал дій** — хто і коли змінював документи, товари, користувачів
5. **Друк** документа (кнопка на картці)

**Що пояснити:** DIRECTOR — read-only для операцій; рішення на основі звітів і аудиту без зміни даних.

---

## Крок 7 — Зони складу (опційно, 30 с)

1. **Товари** → колонка **Зона** (сухий / холод / заморозка)
2. **Партії** або **Залишки** → фільтр по зоні

**Що пояснити:** розділення продовольства за температурними режимами на одному складі.

---

## КВЕД для записки

Для власного складу оптових **продовольчих** товарів доцільно посилатися на **46.39** (оптова торгівля харчовими товарами), а не 46.19 (посередництво без власного складу).

## Обмеження та подальший розвиток

| Реалізовано | Можливе розширення |
|-------------|-------------------|
| Партії, FEFO, 4 типи документів | EDI з постачальниками |
| 3 ролі, JWT | Електронний підпис |
| SQLite / PostgreSQL | Мобільний застосунок для збору |
| CSV + PDF-звіти | Інтеграція з 1С / бухгалтерією |
| Прайс-листи B2B | Автоматичні знижки за обсяг |
| Зони складу (сухий / холод / заморозка) | Штрих-коди, сканери |
| Журнал аудиту дій | Push-сповіщення (email/Telegram) |
| Сповіщення на робочому столі | — |
| Друк документа | — |
